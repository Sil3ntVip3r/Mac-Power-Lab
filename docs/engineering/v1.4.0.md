# MacPowerLab Go Engineering Review

Review target: **MacPowerLab v1.3.0** Go engine
Refactored implementation: **MacPowerLab v1.4.0**
Reviewer perspective: Principal Staff Engineer — Go, clean architecture, and concurrent systems

## Scope and evidence

The review covered `cmd/macpowerlab`, every package under `internal/`, callers,
configuration, storage/report/archive paths, benchmark workloads, local API
lifecycle, schemas, and build documentation. The source archive contains no
`.git` directory and no repository-specific engineering instructions such as
`AGENTS.md` or `CONTRIBUTING.md`; therefore the review used a complete
source-tree diff rather than Git status/history.

Baseline v1.3.0 validation:

```text
gofmt -l cmd internal                    clean
go test -count=1 ./...                   passed
go vet ./...                             passed
go test -race -count=1 ./...             passed
total statement coverage                 42.3%
```

The passing baseline race run does not disprove the findings below: several are
semantic correctness, liveness, stale-data, and error-propagation defects rather
than unsynchronized-memory races, and the existing tests did not exercise all
failure transitions.

## Architectural strengths

- The package boundaries are directionally sound: collection, attribution,
  benchmark control, storage, reports, archives, API, and TUI have distinct
  responsibilities.
- Public JSON/JSONL contracts are versioned and JSONL is a sensible canonical
  persistence format.
- Long-running operations generally accept contexts, and the subprocess layer
  already owns process groups and bounded output capture.
- Consumer-defined interfaces in `benchmark` and `server` are narrow and
  idiomatic; they enable deterministic unit tests without invoking privileged
  macOS tools.
- UI-facing sample/error channels are intentionally bounded and non-blocking,
  which is appropriate for telemetry where the newest value matters more than
  delivering every intermediate frame.
- The API is loopback-oriented, token authenticated, request-size limited, and
  rejects unknown JSON fields.
- The module uses only the Go standard library, reducing dependency and
  supply-chain risk.

# Confirmed defects, ranked by severity

## 1. Battery current and charging-state parsing can corrupt the primary power signal

- **Severity:** Critical
- **Confidence:** High
- **Original locations:**
  - `internal/collector/battery.go:64-81`
  - `internal/collector/battery.go:146-188`
- **Trigger:**
  - Battery current is a legitimate low value such as `50` mA or `-50` mA.
  - `pmset -g batt` contains `discharging`, `charged`, or `not charging`.
- **Failure scenario:** `normalizeCurrent` divided only magnitudes above 100,
  treating `50` mA as `50 A`. Separately, substring matching considered
  `discharging` to contain `charging`.
- **Impact:** Primary load can be wrong by 1,000×, discharge can be reported as
  charging, benchmark power source validation can be wrong, and every downstream
  Wh/runtime/app-attribution calculation can be corrupted.
- **Supporting reasoning:** AppleSmartBattery electrical fields are source-specific
  millivolt/milliamp values. Magnitude heuristics cannot distinguish a low mA
  value from amperes, and state detection must be token/phrase aware.
- **Smallest reasonable fix:** Convert AppleSmartBattery and adapter fields using
  explicit source units; parse active charging only after excluding
  `discharging`, `charged`, and `not charging`.
- **Regression tests:** 12,500 mV + 50 mA = 12.5 V, 0.05 A, 0.625 W; negative
  current preserves sign; 28,000 mV/5,000 mA adapter contract; `discharging`
  never sets charging.
- **v1.4.0:** Fixed in `internal/collector/battery.go` with source-specific
  converters and explicit state parsing.

## 2. The telemetry pipeline can publish stale values, spin on closed channels, and backpressure powermetrics

- **Severity:** High
- **Confidence:** High
- **Original locations:**
  - `internal/collector/manager.go:180-263`
  - `internal/collector/powermetrics.go:55-115`
- **Trigger:** `powermetrics` exits/restarts, process task collection fails, or the
  manager spends longer than one sampler interval collecting battery/storage
  data while the powermetrics output buffer fills.
- **Failure scenario:**
  - Closed `pmCh`/`pmErr` channels remained permanently ready in `select`, causing
    a tight loop.
  - The last successful component/process values were reused indefinitely with no
    collection timestamp or staleness policy.
  - A blocking producer send could stop reading the child stdout pipe while the
    consumer was temporarily busy.
- **Impact:** High CPU usage after sampler failure, apparently live but stale CPU,
  GPU, thermal, and app readings, and possible child-process backpressure.
- **Smallest reasonable fix:** Nil closed channels, attach freshness timestamps,
  expire data after a bounded multiple of the configured interval, expose source
  status, and use a one-element latest-value replacement channel.
- **Regression tests:** Freshness-window boundaries, closed-channel handling,
  stale-process clearing, and replacement of an already-buffered snapshot.
- **v1.4.0:** Fixed. Optional app attribution is also reported as `disabled`
  rather than falsely `unavailable`.

## 3. Application power attribution mishandles present zero rates and biases CPU over GPU

- **Severity:** High
- **Confidence:** High
- **Original locations:**
  - `internal/collector/powermetrics.go:425-446`
  - `internal/attribution/attribution.go:128-130`
- **Trigger:**
  - `energy_impact_per_s` is present and legitimately zero while raw
    `energy_impact` is non-zero.
  - Reported CPU+GPU component power exceeds the measured dynamic-power pool.
- **Failure scenario:** A present zero rate was interpreted as missing and
  replaced with the raw value. Component allocation gave CPU the first claim on
  the pool and GPU only the remainder.
- **Impact:** Idle apps can receive artificial residual weight, and app CPU/GPU
  estimates depend on field ordering rather than measured component ratios.
- **Smallest reasonable fix:** Return `(value, present)` from rate extraction;
  preserve present zero. If component estimates exceed dynamic power, scale CPU
  and GPU proportionally before allocating to apps.
- **Regression tests:** Present-zero rate fixture and a 30 W CPU + 30 W GPU input
  constrained to a 40 W dynamic pool, expecting 20 W/20 W and exact conservation.
- **v1.4.0:** Fixed with presence-aware extraction and proportional pool scaling.

## 4. Benchmark execution is not internally single-owner and can lose or overwrite evidence

- **Severity:** High
- **Confidence:** High
- **Original locations:**
  - `internal/benchmark/benchmark.go:153-280`
  - `internal/benchmark/benchmark.go:412-537`
- **Trigger:** Two callers invoke `Controller.Run` concurrently, preparation fails
  before progress is published, or the same phase name is run more than once.
- **Failure scenario:** The HTTP engine attempted single-run ownership, but the
  controller itself did not enforce it. Preflight failures left stale progress;
  `TestRun.Commands` was empty; workload logs used phase-name filenames with
  `O_TRUNC`, so later runs overwrote earlier evidence.
- **Impact:** Duplicate heavy workloads can run concurrently, progress streams can
  interleave, and diagnostic logs/commands can be lost—the exact evidence needed
  to explain a power or thermal anomaly.
- **Smallest reasonable fix:** Enforce ownership inside the controller, publish a
  `preparing` state before fallible preflight, derive terminal status from the
  returned error, generate run-unique directories, and persist the actual command
  argv.
- **Regression tests:** Concurrent `Run` rejection, preparation failure status,
  unique run IDs/log paths, and command persistence.
- **v1.4.0:** Fixed. Workload cleanup, log sync/close, and copy errors are now
  returned instead of discarded.

## 5. Optional SQLite mirror failures repeatedly poison otherwise successful JSONL writes

- **Severity:** High
- **Confidence:** High
- **Original location:** `internal/store/store.go:171-249`
- **Trigger:** `sqlite3` starts successfully but later exits, its pipe fails, or a
  flush returns an error.
- **Failure scenario:** Canonical JSONL was already written, but every later
  sample/event/test write retried the broken optional mirror and returned an
  error.
- **Impact:** Error floods, repeated failed work on every sample, and the monitor
  appears unhealthy despite its canonical data being intact.
- **Smallest reasonable fix:** On the first runtime mirror failure, close and
  disable the mirror atomically, return one typed degradation error, and continue
  JSONL-only operation.
- **Regression test:** Inject a mirror that fails once; verify one degradation
  error, no retries, and subsequent JSONL writes succeed.
- **v1.4.0:** Fixed through the `sqliteMirror` abstraction and
  `MirrorDegradedError`.

## 6. CLI commands ignore monitor shutdown failures and can report/archive incomplete data

- **Severity:** High
- **Confidence:** High
- **Original location:** `cmd/macpowerlab/main.go:132,170,295`
- **Trigger:** Manager shutdown times out, flushing fails, session metadata cannot
  be finalized, or workload/TUI cleanup remains active.
- **Failure scenario:** The CLI used `_ = m.Stop(stopCtx)` and immediately generated
  reports or archives.
- **Impact:** A command can exit successfully with incomplete JSONL buffers,
  missing session end metadata, or an archive created while writers are still
  active.
- **Smallest reasonable fix:** Propagate/aggregate stop and TUI errors; do not
  generate reports or archives until stop succeeds.
- **Regression test:** Inject a stop timeout/failure and assert report/archive
  generation is not invoked.
- **v1.4.0:** Fixed in monitor, app-audit, and benchmark command paths.

## 7. Parity compares different time windows

- **Severity:** Medium
- **Confidence:** High
- **Original location:** `internal/parity/parity.go:47-64`
- **Trigger:** System load or charger/battery state changes during an iteration.
- **Failure scenario:** The Go collector completed before the legacy collector was
  started.
- **Impact:** Differences can reflect time skew rather than implementation parity,
  producing false failures or hiding real ones.
- **Smallest reasonable fix:** Launch both collectors concurrently under the same
  parent context and compare after both finish.
- **Regression test:** Block both fake collectors and assert each starts before
  either is released.
- **v1.4.0:** Fixed with `collectPair`.

## 8. XML plist parsing has no structural resource limits or strict root validation

- **Severity:** Medium
- **Confidence:** High
- **Original location:** `internal/plist/plist.go:16-168`
- **Trigger:** Deeply nested, extremely wide, duplicate-key, multi-root, or
  malformed plist input from imported diagnostics or a changed tool output.
- **Failure scenario:** Recursive parsing had no depth/node limit; duplicate keys
  silently overwrote previous values; the parser returned after the first root
  without validating trailing elements/data.
- **Impact:** Stack/memory exhaustion, ambiguous sensor selection, and acceptance
  of malformed machine-readable input.
- **Smallest reasonable fix:** Bound nesting and node count, reject duplicate or
  dangling dictionary keys, require exactly one root value, and reject non-space
  trailing content.
- **Regression tests:** Excessive depth, duplicate keys, multiple roots, dangling
  keys, and unexpected character data.
- **v1.4.0:** Fixed with a bounded stateful parser.

## 9. Reports omit collector failures and accept cross-session or out-of-order samples

- **Severity:** Medium
- **Confidence:** High
- **Original location:** `internal/report/report.go:51-112`
- **Trigger:** A session contains collector-error events, a sample from another
  session, an unsupported schema, or a timestamp regression.
- **Failure scenario:** Reports consumed samples without checking schema/session or
  ordering and never read `events.jsonl` for collector failures.
- **Impact:** A polished report can hide sensor degradation, integrate invalid
  data, or silently combine records from different sessions.
- **Smallest reasonable fix:** Validate sample schema, session ID, and monotonic
  timestamps; merge `collector_error`/warning events into report warnings.
- **Regression tests:** Mismatched session, backwards timestamp, unsupported
  schema, and collector-error event inclusion.
- **v1.4.0:** Fixed.

## 10. Archive file validation has a check/open race

- **Severity:** Medium
- **Confidence:** High
- **Original location:** `internal/archive/archive.go:170-208`
- **Trigger:** A regular file is replaced with a symlink or another inode after
  `Lstat` and before `Open`.
- **Failure scenario:** The archive validated one path object but streamed a later
  object opened by name.
- **Impact:** An archive can include content that was not the file originally
  validated and listed.
- **Smallest reasonable fix:** Open, `fstat`, require regular mode, and verify
  `os.SameFile` against the pre-open identity before writing the header/content.
- **Regression test:** Replace the path after capturing expected metadata and
  assert identity mismatch rejection.
- **v1.4.0:** Fixed with `openExpectedRegular`.

## 11. The local API trusts the requested host more than the actual listener and splits shutdown ownership

- **Severity:** Medium
- **Confidence:** High
- **Original location:** `internal/server/server.go:437-478,586-617`
- **Trigger:** Hostname resolution binds unexpectedly, or `Serve` exits for a
  reason other than the normal context cancellation path.
- **Failure scenario:** Only the input string was checked for loopback. Shutdown
  was initiated in one goroutine while deferred engine cleanup independently
  owned related lifecycle work.
- **Impact:** Defense-in-depth around loopback binding was incomplete, and shutdown
  errors/order were difficult to reason about.
- **Smallest reasonable fix:** Validate the actual `TCPAddr` after `net.Listen`,
  establish one server-shutdown owner, cancel it if `Serve` exits, and join
  shutdown/serve/engine-close errors.
- **Regression tests:** Loopback accepted, wildcard rejected, context cancellation,
  and unexpected `Serve` return.
- **v1.4.0:** Fixed.

## 12. Small hot-path allocations are avoidable, but do not justify `sync.Pool`

- **Severity:** Low
- **Confidence:** High
- **Original locations:**
  - `internal/collector/manager.go:345-355`
  - `internal/collector/helpers.go:114-159`
- **Trigger:** Every one-second temperature sample and repeated lookup of stable,
  exact macOS plist keys.
- **Failure scenario:** Temperature pruning allocated a fresh slice every sample;
  exact-key lookups sorted the full map before checking the common exact spelling.
- **Impact:** Avoidable small allocation/CPU overhead—not a correctness or current
  GC risk.
- **Smallest reasonable fix:** Prune the retained slice in place and use an exact
  map lookup fast path before deterministic case-insensitive fallback.
- **Regression test:** Existing collector benchmarks plus functional lookup tests.
- **v1.4.0:** Fixed without introducing pooling.

# Idiomatic Go, package organization, and testability

## What is idiomatic

- Narrow interfaces live with consumers rather than being imposed globally.
- Error wrapping generally preserves causal context with `%w` and `errors.Join`.
- Context ownership is now explicit for monitor, benchmark, subprocess, and API
  lifetimes.
- Buffered channels are used for status/telemetry where dropping obsolete values
  is preferable to blocking collectors.
- Storage resources have clear ownership and idempotent close behavior.

## Optional structural improvements

These are not confirmed correctness defects and were not forced into the
compatibility-preserving refactor:

1. `cmd/macpowerlab/main.go` has become large. Moving flag parsing and each command
   into `internal/cli` would improve navigation and unit coverage.
2. Battery command execution is concrete while parsing is pure. Introducing a
   very small command-runner interface could make optional-source failure tests
   easier, but should not become a broad abstraction hierarchy.
3. The current lower-envelope baseline intentionally resists upward drift during
   stress. A future model may need controlled upward adaptation after hours of
   genuine environmental change; that requires product semantics, not a local
   code cleanup.
4. Numeric v1 fields use zero for both “unavailable” and legitimate zero. Correct
   presence semantics require pointers/validity fields and a versioned v2 schema.

# Performance and memory assessment

Benchmarks on Linux/amd64, Go 1.23.2:

```text
BenchmarkAttributorObserve200Processes  167574 ns/op  151115 B/op  213 allocs/op
BenchmarkParsePowermetricsFixture        10600 ns/op    4564 B/op   59 allocs/op
```

The attributor normally runs every 10 seconds, so its measured allocation rate is
about 15 KiB/s. This is not a meaningful garbage-collector bottleneck. The
important memory/liveness controls are bounded process rows, bounded plist
frames/output, stale-value expiry, and bounded accumulated app state.

A `sync.Pool` is **not recommended** now. Pooling large plist buffers or maps would
retain peak allocations, complicate ownership, and create stale-data risks. Add
pooling only after a real on-Mac heap/CPU profile identifies a stable high-rate
allocation hot spot.
